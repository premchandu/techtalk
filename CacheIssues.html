Engineering Deep Dive

When Your Cache Breaks the System

A comprehensive guide to every major caching failure mode — what causes them, how they cascade, and how to defend against them.

Contents Penetration Avalanche Breakdown Hotspot Inconsistency Stampede Pollution Overflow Summary

Caching is the cornerstone of every high-throughput system. Done right, it slashes latency and keeps your database alive under load. Done wrong, it becomes the single most catastrophic point of failure in your architecture — one that can take down a database cluster in seconds or serve stale data to millions of users without anyone noticing for hours.

Beyond the famous trio of penetration, avalanche, and breakdown, there are five more failure modes that production systems regularly encounter. This guide covers all eight — from their root causes and blast radius to every known mitigation strategy, with real implementation examples.

01
⬤ High Severity

Cache Penetration

Requests that bypass the cache entirely, every single time

Cache penetration occurs when queries are made for data that does not exist in either the cache or the database. Because there is no result to cache, every repeated query pierces straight through to the database — the cache provides zero protection.

This is a particularly dangerous failure mode because it can be deliberately triggered. An attacker who discovers that querying user_id=-1 always misses the cache can flood your system with millions of such requests, effectively using your cache as a weapon against your own database.

Real World Scenario
An e-commerce site receives millions of requests for product IDs that don't exist (bots enumerating IDs, broken links, or malicious probing). The cache misses every time. The database takes the full load and falls over.
Request flow — penetration pattern
1
Request arrives for non-existent key e.g. GET /user?id=-99999 or a malformed UUID
2
Cache MISS Key doesn't exist in Redis/Memcached — expected, not cached because it's never valid
3
Database query executes Returns NULL / empty result — no record exists
✗
Nothing is cached — cycle repeats infinitely Next identical request hits the DB again. Under load this destroys the database.
Solutions & Defences
Cache Null Results Primary

When a DB query returns nothing, store a sentinel value (e.g. "NULL") with a short TTL (30–300s). Subsequent requests hit the cache and return fast. Be careful: if the data is later created, the stale null must be invalidated.

Bloom Filter Primary

A probabilistic data structure that answers "definitely not in the set" with zero false negatives. Put all valid keys into a Bloom filter at startup. Requests for keys the filter rejects never reach the DB at all.

Input Validation Complementary

Validate ID formats and ranges before any cache/DB lookup. Negative IDs, malformed UUIDs, or IDs beyond the max sequence should return 400 immediately.

Rate Limiting Complementary

Apply per-IP and per-key request rate limits at the edge layer. Burst protection prevents volumetric penetration attacks even if they slip past other filters.

// Solution: Cache null results with short TTL
async function getUser(id) {
  const cacheKey = `user:${id}`;
  const cached = await redis.get(cacheKey);

  if (cached === "__NULL__") return null; // sentinel hit
  if (cached) return JSON.parse(cached);

  const user = await db.findUser(id);

  if (!user) {
    await redis.setex(cacheKey, 60, "__NULL__"); // 60s TTL
    return null;
  }

  await redis.setex(cacheKey, 3600, JSON.stringify(user));
  return user;
}
02
⬤ Critical Severity

Cache Avalanche

Mass cache expiration or cache node failure causing a DB tsunami

Cache avalanche describes any event where a large portion of cached data expires simultaneously — or an entire cache tier goes down — causing a sudden, enormous flood of requests to hit the database all at once. Unlike penetration (targeted) or breakdown (single hot key), avalanche is a systemic collapse event.

The two classic triggers are: (1) mass TTL expiration — all cache entries were populated at the same time during a cold start, so they all expire at the same time; and (2) cache server failure — the Redis cluster crashes, taking all cached data with it.

Blast radius: Avalanche events typically cause full database outages within 30–90 seconds. Even well-provisioned databases can be overwhelmed because the query volume increases by 10x–100x instantly, with no warm-up period.
Timeline of an avalanche event
T=0
Cold start / cache restart 10,000 cache keys are populated over 2 minutes. All are given TTL = 3600s.
T=1h
Mass expiration begins All 10,000 keys expire within a 2-minute window — because they were all set at the same time.
T=1h+30s
Database overwhelmed Every incoming request is a cache miss. DB connection pool exhausted. Query queue grows unbounded.
T=1h+90s
Cascade failure DB response times spike → app timeouts → retry storms amplify the load → full outage.
Solutions & Defences
TTL Jitter Primary

Randomize TTL values within a range: TTL = base + random(0, jitter). This spreads expiration events across time, preventing synchronized mass expiry. Simple and highly effective.

Cache HA / Clustering Primary

Redis Sentinel or Redis Cluster with replica promotion eliminates single points of failure. When a master fails, a replica is promoted within seconds with data intact.

Circuit Breaker Complementary

When cache miss rate spikes above a threshold, open a circuit breaker that serves degraded/cached responses rather than hammering the DB. Let the cache rebuild before re-enabling full queries.

Multi-Level Cache Complementary

Layer L1 (in-process, e.g. Caffeine) + L2 (Redis). If Redis fails, L1 continues serving recently-accessed data. L1 TTLs are typically very short (seconds to minutes), reducing DB exposure.

// TTL jitter prevents synchronized expiration
function setWithJitter(redis, key, value, baseTTL, jitter = 300) {
  const ttl = baseTTL + Math.floor(Math.random() * jitter);
  return redis.setex(key, ttl, value);
}

// e.g. base 3600s ± up to 300s → expires 60–65 min from now
setWithJitter(redis, "product:123", data, 3600, 300);
03
⬤ Critical Severity

Cache Breakdown

A single hot key expires under concurrent load — the thundering herd

Cache breakdown (also called thundering herd or cache stampede) occurs when a single, extremely popular cache key expires at the exact moment it is being read by many concurrent requests. Since none of them find the key in the cache, all of them simultaneously query the database and attempt to repopulate the cache — multiplying the load by the number of concurrent requesters.

This is common for content like homepage banners, trending product pages, or live sports scores — data that is accessed by thousands of users per second and is expensive to recompute from the database.

Breakdown: 1000 concurrent requests, hot key just expired
✗
Hot key expires (e.g. trending_products) 1,000 requests in flight all miss the cache at the same instant
✗
All 1,000 query the DB simultaneously Each executes the same expensive JOIN query. DB CPU spikes to 100%.
~
All 1,000 attempt to write the result to cache Race condition — each overwriting the other. Cache network gets flooded too.
Solutions & Defences
Mutex Lock (SETNX) Primary

When a cache miss is detected, only one thread acquires a distributed lock (via Redis SETNX) and queries the DB. Other threads wait briefly, then get the newly-populated cache entry.

Logical Expiration Primary

Store an embedded expiry timestamp inside the value itself. Never let the key truly expire in Redis. Background threads refresh proactively before the logical expiry is reached. All reads always get a value (possibly slightly stale).

Probabilistic Early Refresh Advanced

XFetch algorithm: as a key approaches its TTL, each request has an increasing probability of triggering a proactive refresh. No lock needed. Distributes the refresh work naturally.

Stale-While-Revalidate Complementary

Serve the stale (expired) value immediately while one background process refreshes the cache. Works well when brief staleness is acceptable and read performance is critical.

// Mutex lock pattern — only one DB query for a hot miss
async function getHotData(key) {
  let value = await redis.get(key);
  if (value) return JSON.parse(value);

  const lockKey = `lock:${key}`;
  const acquired = await redis.set(lockKey, "1", "NX", "EX", 3);

  if (acquired) {
    // We won the race — do the DB query
    const data = await db.expensiveQuery(key);
    await redis.setex(key, 3600, JSON.stringify(data));
    await redis.del(lockKey);
    return data;
  } else {
    // Lost the race — wait briefly then re-read
    await sleep(50);
    return getHotData(key); // retry — should hit cache now
  }
}
04
⬤ High Severity

Cache Hotspot (Skewed Access)

Traffic concentrates on a tiny fraction of keys, overloading a single node

In distributed cache clusters, keys are partitioned across nodes (by consistent hashing or slot assignment). A hotspot emerges when a very small number of keys — or even a single key — receive a disproportionate share of traffic, overwhelming the one node responsible for them while other nodes sit idle.

This is different from breakdown (it's about sustained load, not expiration) and different from avalanche (it's localized, not global). A celebrity posting on a social platform, a viral product link, or a live streaming event can create a hotspot that degrades an entire cluster due to one overloaded Redis shard.

Solutions & Defences
Key Replication / Mirroring Primary

Copy hot keys to N replica shards using a numeric suffix: hot_key:0, hot_key:1, ..., hot_key:N. Requests are randomly routed to any replica, spreading load across N nodes.

Local In-Process Cache (L1) Primary

Each application server maintains a small in-memory cache. Hot keys live there for a few seconds, absorbing per-server traffic before it ever reaches Redis. L1 misses fall through to Redis.

Read-Through with Coalescing Complementary

Deduplicate in-flight requests for the same key at the application layer. If 100 requests arrive simultaneously for the same hot key, coalesce them into one upstream call and fan the result back to all 100 waiters.

Hotspot Detection & Alerting Operational

Monitor per-key QPS in your Redis layer. Alert when a single key exceeds a threshold (e.g. 10k ops/sec). Automated hot-key detection is available in Redis Enterprise and AWS ElastiCache.

05
⬤ Medium Severity

Cache Inconsistency (Dirty Cache)

The cache and database diverge — users get stale or wrong data

Cache inconsistency occurs when data in the cache no longer matches the data in the source of truth (the database). This can happen through update race conditions, failed cache invalidations, partial write failures, or failure to invalidate related keys when one key changes.

This is unique among cache failures because it doesn't cause performance problems — it silently serves wrong data. Users may see outdated prices, incorrect inventory counts, or deleted content. In financial or healthcare systems this can have serious consequences.

Race condition — stale write wins
1
Thread A reads x=1 from DB Cache miss, proceeds to read from DB
2
Thread B updates x=2, invalidates cache DB now has x=2, cache is empty
3
Thread A writes x=1 to cache (stale!) Cache now shows x=1 even though DB has x=2. Inconsistency introduced.
Cache Write Strategies
Strategy How It Works Consistency Performance
Cache-Aside App reads cache, on miss reads DB and populates cache Eventual Good
Write-Through Every DB write also updates cache synchronously Strong Write overhead
Write-Behind Write to cache first, async DB write follows Weak Excellent
Refresh-Ahead Proactively refresh cache before TTL expires Good Good
Cache Invalidation Delete cache key on every write; lazy repopulation Strong Miss penalty
Advanced Techniques
Version Tags / Cache Versioning Primary

Include a version token in cache keys: user:123:v42. On invalidation, bump the version globally. Old keys are ignored and naturally expire. Eliminates stale reads without explicit deletion.

Change Data Capture (CDC) Primary

Tools like Debezium stream DB write-ahead log changes to a message broker (Kafka). Cache invalidation listeners consume these events and purge/update affected keys in near-real-time.

06
⬤ High Severity

Cache Stampede (Dog-Pile)

After a cache flush or cold start, all requests pile onto an empty cache

A cache stampede is a variation of breakdown that occurs when the entire cache is cold — immediately after a deployment, cache flush, or service restart. Unlike breakdown (a single key expiring), stampede involves the whole cache being empty and traffic slamming into the database from the moment the service starts.

Stampedes are predictable events (they happen at every deploy) but are often overlooked in capacity planning. A system that handles normal traffic fine can completely fail for the first 60 seconds after a restart.

Solutions & Defences
Cache Warming Primary

Before routing production traffic to a new instance, run a warm-up script that pre-populates the top-N most frequently accessed keys from the DB or a data snapshot. Service opens only when warm.

Gradual Traffic Ramp Primary

Use traffic splitting: route 5% of traffic to the new instance first. The cache warms under controlled load. Increase gradually to 100% as hit rate recovers. Blue-green and canary deployments facilitate this.

Shadow Cache Persistence Complementary

Periodically snapshot the Redis cache to disk (RDB files). On restart, reload from the snapshot. Cache is immediately warm with data from minutes ago rather than being completely empty.

Request Queuing / Coalescing Complementary

During warm-up, queue duplicate requests for the same key. Only one DB query is made per key; the result fans out to all waiters. Prevents the "1000 threads, 1 key" problem during cold periods.

07
⬤ Medium Severity

Cache Pollution

Infrequently-accessed data evicts high-value data from a limited cache

Cache pollution occurs when low-value or one-time-accessed data fills the cache, evicting high-frequency data that would have delivered much greater hit-rate value. The classic example: a bulk data export or a search scan reads millions of records once, each landing in the cache, flushing out your hot user profiles and product pages.

Symptoms are subtle: hit rate gradually degrades, average latency slowly rises, and there's no obvious error to trace. It's often diagnosed late by comparing hit rates before and after a large batch operation.

Solutions & Defences
LFU Eviction Policy Primary

Least-Frequently-Used eviction naturally protects hot keys. Redis supports allkeys-lfu. Items accessed once are evicted long before those accessed thousands of times, regardless of recency.

Separate Cache Namespaces Primary

Use distinct Redis databases or separate Redis instances for different data categories (hot user data vs. batch/export data). Pollution in the batch namespace cannot affect the hot namespace.

TTL Tiering by Access Pattern Complementary

Assign very short TTLs (seconds) to data accessed rarely or in bulk scans. Assign long TTLs (hours) to hot user-facing data. Low-value data expires quickly before it can evict high-value data.

Bypass Cache for Batch Ops Operational

Route bulk exports, reporting queries, and admin batch reads directly to a read-replica DB, completely bypassing the cache layer. Preserves cache for user-facing traffic.

08
⬤ Medium Severity

Cache Overflow & Memory Pressure

Cache memory fills up — eviction policies determine what breaks

Every cache has a finite memory limit. When that limit is reached, the configured eviction policy determines what happens next — and a wrong choice here can cause behavior ranging from silent performance degradation to outright errors. In Redis, the default noeviction policy causes write errors when memory is full, which is often the worst possible behavior for a production cache.

Beyond eviction policies, memory fragmentation — where allocated but unused memory gaps accumulate — can effectively reduce usable cache space without the memory limit being reached, causing subtle degradation over days or weeks.

Redis Policy Behavior at Max Memory Best For
noeviction Returns error on new writes Avoid for caches
allkeys-lru Evicts least recently used from all keys General cache
allkeys-lfu Evicts least frequently used from all keys Hot key protection
volatile-lru Evicts LRU among keys that have a TTL set Mixed TTL workloads
allkeys-random Evicts random key from all keys Uniform access only
Solutions & Defences
Use allkeys-lfu or allkeys-lru Primary

Change the default noeviction policy to allkeys-lfu for any Redis instance used purely as a cache. This allows graceful eviction rather than write failures.

Memory Monitoring & Alerts Primary

Alert at 75% and 90% memory usage. At 75%, trigger capacity review. At 90%, trigger automatic scale-out (horizontal Redis Cluster expansion or new shards).

Active Defragmentation Complementary

Enable Redis activedefrag yes. Redis will gradually compact memory pages in the background, recovering fragmented space without a restart.

Value Compression Complementary

Compress large cached objects (msgpack, snappy, zstd) before storing. A 10KB JSON value compressed with zstd may drop to 1–2KB. Effectively multiplies cache capacity with minimal CPU overhead.

At a Glance

Eight failure modes, each with a distinct fingerprint. The three most dangerous — avalanche, breakdown, and penetration — share one solution philosophy: prevent synchronized, uncontrolled database access. The rest require careful attention to data freshness, memory discipline, and deployment hygiene.
